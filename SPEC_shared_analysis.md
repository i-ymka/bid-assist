# SPEC: Общий кэш AI-анализа между аккаунтами

## Суть

Когда два аккаунта (yehia + ymka) находят один и тот же проект, они оба запускают Gemini Call 1 (feasibility analysis) независимо — это двойная трата токенов на идентичный результат. Нужно запускать Call 1 ровно один раз на проект, а Call 2 (написание бида) оставить раздельным — у аккаунтов разные персоны.

---

## Что входит в скоуп

1. **Общий кэш Call 1** — отдельный файл `data/shared_analysis.db` (или таблица в одном из DB), доступный обоим процессам. Хранит: `project_id`, `verdict`, `days`, `summary`, `status` (pending/done/skip), `created_at`.

2. **Проверка перед Call 1** — перед запуском `analyze_feasibility()` каждый аккаунт проверяет общий кэш. Если запись с `status=done` или `status=skip` существует — использует готовый результат, Call 1 не запускается.

3. **Блокировка гонки** — если запись в кэше есть но `status=in_progress` (другой аккаунт уже анализирует) — текущий аккаунт ждёт или пропускает проект до следующего цикла (не запускает второй параллельный Call 1).

4. **Call 2 остаётся раздельным** — каждый аккаунт запускает `write_bid()` самостоятельно со своей persona, `bid_adjustment`, `min_daily_rate`. Кэш Call 1 только передаёт в него `verdict`, `days`, `summary`.

5. **Polling остаётся раздельным** — каждый аккаунт опрашивает Freelancer API независимо (могут быть разные skill_ids, бюджеты, фильтры). Один и тот же проект может попасть в очередь обоих, и это нормально — Call 1 всё равно выполнится один раз.

6. **SKIP распространяется на оба аккаунта** — если Call 1 вернул SKIP (проект нецелесообразен с технической точки зрения), оба аккаунта пропускают его. VERDICT — это оценка проекта, а не аккаунта.

7. **TTL кэша** — записи в общем кэше живут не более 24 часов. Устаревшая запись игнорируется, Call 1 запускается заново.

---

## Что НЕ входит (явно)

- Объединение процессов yehia + ymka в один — каждый остаётся отдельным subprocess с изолированным Settings-singleton.
- Общая очередь `project_queue` — у каждого аккаунта своя очередь, своя БД.
- Общая история ставок, настройки, bid_outcomes — всё per-account.
- Дедупликация polling на уровне API (оба опрашивают Freelancer независимо).
- Логика "если yehia пропустил — ymka тоже пропускает фильтрацию" — фильтры раздельные.
- Sharing результатов Call 2 (bid текст) — у каждого аккаунта своя persona.

---

## Критерии готовности

1. Запустить обоих ботов. Когда один и тот же проект попадает в очередь обоих — в логах видно ровно одну строку вида `"Call 1: analyzing project 12345"` и одну строку `"Call 1: using cached result for project 12345"`. Вторая строка `"Call 1: analyzing project 12345"` не появляется.

2. В `data/shared_analysis.db` есть запись для проекта с `status=done`, `verdict`, `days`, `summary` заполнены.

3. Оба аккаунта получают BID-уведомление в Telegram по этому проекту (если VERDICT=PASS), но тексты бидов разные (разные персоны).

4. Если проект попадает только к одному аккаунту (навыки не пересекаются) — Call 1 запускается нормально, кэш записывается, второй аккаунт этот проект не видит и не затрагивает.

5. Гонка: если оба аккаунта обнаружили проект одновременно — только один из них запускает Call 1 (статус `in_progress` блокирует второго). Второй использует результат после завершения. В логах НЕТ двух одновременных вызовов Gemini для одного project_id.

6. После 24 часов: запись в кэше устарела. Следующий аккаунт, который видит проект в очереди, запускает Call 1 заново и обновляет кэш.

---

## Решённые вопросы (зафиксировано)

1. **Где хранить общий кэш** → отдельный `data/shared_analysis.db`. Нет зависимости между аккаунтами на уровне файлов. Если запущен один аккаунт — файл просто создаётся пустым, overhead нулевой. Флаг включения не нужен.

2. **Зависший `in_progress`** → тот же паттерн что уже работает в `project_queue`: `cleanup_loop` (уже существует) чистит записи старше 24ч. Добавить `shared_analysis` с `status='in_progress'` к той же очистке. Отдельного решения не требуется.

3. **SQLite concurrency** → `INSERT OR IGNORE` + `UPDATE WHERE status='pending'` для атомарного захвата. WAL mode включается при создании shared DB (`PRAGMA journal_mode=WAL`).
